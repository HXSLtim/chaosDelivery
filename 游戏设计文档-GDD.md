# 《混乱快递》游戏设计文档 (GDD)

> **游戏类型**：派对合作游戏  
> **平台**：Steam (支持局域网 + Steam好友联机)  
> **人数**：2-4人  
> **单局时长**：10-15分钟  
> **美术风格**：Low Poly + 明亮卡通色块

---

## 一、核心概念

### 1.1 一句话描述
> 2-4名玩家扮演快递站员工，在限时内协作分拣、打包、装车，应对各种突发混乱，完成越来越离谱的订单。

### 1.2 核心体验支柱

| 支柱 | 描述 | 设计决策检验标准 |
|------|------|------------------|
| **混乱中的配合** | 玩家在混乱环境下必须沟通协作 | 每个关卡都有迫使玩家说话的设计 |
| **简单上手，深度配合** | 单个动作极简单，组合后产生策略 | 新手30秒内掌握基础，高手追求最优路线 |
| **搞笑失误 > 失败惩罚** | 失误有趣而非挫败 | 没有"游戏结束"，只有"搞砸了但很好笑" |
| **好友开黑优化** | 为熟人社交设计，支持语音/线下同屏 | 支持Steam远程同乐 + 本地分屏 |

### 1.3 参考游戏
- **Overcooked**：时间压力 + 多任务协作
- **Moving Out**：物理搬运 + 搞笑混乱
- **Unrailed**：共同目标 + 即时沟通需求

## 二、核心游戏循环

### 2.1 即时循环（0-5秒）
```
识别需求 → 执行动作 → 获得反馈 → 调整下一步
```

| 动作 | 输入 | 反馈 | 调整 |
|------|------|------|------|
| 抓取包裹 | 空格键 | 角色双手举起，包裹摇晃 | 决定搬运路线 |
| 放置/投掷 | 松开空格 | 包裹落地/入车音效 | 检查是否正确 |
| 扫描标签 | 对准+点击 | "滴"声 + 绿色勾 | 确认目的地 |
| 操作设备 | 长按交互键 | 进度条 + 机械音效 | 等待或放弃 |

### 2.2 会话循环（单局 10-15分钟）

```
准备阶段 (30s) → 工作阶段 (8-12min) → 结算阶段 (1-2min)
```

#### 准备阶段
- 查看今日订单列表（3-5个订单）
- 分配角色/区域（自发协商，无强制）
- 观察关卡特殊机制（随机事件预告）

#### 工作阶段
```
订单生成 → 分拣包裹 → 处理特殊需求 → 装车发货 → 新订单
     ↑___________________________________________↓
```

**订单流程**：
1. **订单生成**：屏幕侧边弹出订单卡片（目的地 + 包裹类型 + 特殊要求）
2. **分拣包裹**：从传送带/货架找到对应包裹
3. **处理需求**：
   - 易碎品：轻拿轻放，不能投掷
   - 急件：优先处理，倒计时更短
   - 大件：需2人搬运
   - 危险品：不能碰撞，需特殊通道
4. **装车发货**：将包裹放入对应区域的货车
5. **完成奖励**：金币 + 星级评价

#### 结算阶段
- 显示本局数据：完成订单数、准确率、最佳配合时刻
- 获得金币和经验
- 解锁进度提示

### 2.3 长期循环（跨局进度）

```
完成关卡 → 获得金币/经验 → 解锁新内容 → 挑战更高难度 → 完成关卡
```

**解锁树**：
- **新关卡**：不同主题的快递站（港口、机场、太空站）
- **新机制**：新增包裹类型、设备、障碍物
- **装扮**：角色皮肤、表情动作
- **挑战模式**：时间挑战、完美运行、极限难度

## 三、核心机制设计

### 3.1 包裹系统

| 包裹类型 | 视觉特征 | 处理方式 | 失败后果 |
|----------|----------|----------|----------|
| **普通包裹** | 棕色纸箱 | 常规搬运 | 无 |
| **易碎品** | 带玻璃杯图标 | 不能投掷，掉落即碎 | 扣金币，需重新取货 |
| **急件** | 红色闪电标记 | 优先处理，倒计时短 | 超时扣大量金币 |
| **大件** | 体积大，带重量标识 | 移速-40%，可单人搬运 | 速度受限，影响效率 |
| **危险品** | 骷髅/火焰图标 | 走专用通道，不能碰撞 | 爆炸，清空周围包裹 |
| **活体** | 会动/有声音 | 会逃跑，需追赶抓取 | 逃跑后重新生成 |

### 3.2 关卡机制

**基础元素**：
- 传送带：持续生成包裹
- 分拣台：扫描后按目的地分类
- 货架：临时存储
- 装车区：货车等待，装满后发车

**动态事件系统**（详细规格）：

#### 事件触发规则
| 参数 | 数值 | 说明 |
|------|------|------|
| 最小触发间隔 | 45秒 | 防止事件过于密集 |
| 最大同时事件数 | 2个 | 避免玩家 overwhelm |
| 事件持续时间 | 15-30秒 | 短暂但影响显著 |
| 触发概率曲线 | 随时间递增 | 开局10% → 终局40% |

#### 事件详情表

| 事件 | 效果 | 持续时间 | 触发权重 | 应对策略 |
|------|------|----------|----------|----------|
| **传送带加速** | 包裹生成速度×2 | 20秒 | 25% | 临时增加分拣人手 |
| **系统故障** | 扫描机失灵 | 15秒 | 20% | 人工记忆目的地 |
| **货车提前** | 当前货车30秒后发车 | - | 15% | 紧急装车，放弃部分订单 |
| **特殊订单** | 插入高奖励紧急订单 | 订单存在期间 | 20% | 评估是否切换优先级 |
| **障碍物** | 临时出现阻挡路线 | 25秒 | 15% | 绕路或协作清除 |
| **停电** | UI遮罩模拟黑暗，视野缩小50% | 20秒 | 5% | 依靠记忆和语音沟通 |

#### 事件组合限制
- 禁止同时触发：传送带加速 + 系统故障（过于困难）
- 推荐组合：传送带加速 + 特殊订单（高风险高回报）
- 事件冷却：同一事件触发后，60秒内不会再次触发

### 3.3 物理系统

**设计原则**：可控的混乱

- 角色碰撞：玩家可以互相推挤（增加喜剧效果）
- 包裹物理：投掷有抛物线，可反弹
- 连锁反应：包裹碰撞可能引发倒塌

#### 关键参数（初始测试值）

| 参数 | 初始值 | 测试范围 | 设计理由 | 手感目标 |
|------|--------|----------|----------|----------|
| 角色移动速度 | 4.5 m/s | 3.5-6.0 | 略快于步行，有紧迫感 | 能快速穿越中等房间 |
| 角色转身速度 | 360°/s | 300-450° | 即时响应，无惯性 | 操作跟手，无延迟感 |
| 投掷距离（最小） | 2.5 m | 2.0-3.5 | 短距离精准投掷 | 可扔到隔壁工位 |
| 投掷距离（最大） | 7.0 m | 6.0-9.0 | 长距离需要蓄力 | 可扔到房间对角 |
| 蓄力时间 | 1.2秒 | 0.8-1.5 | 有策略性但不拖沓 | 感觉"刚好够瞄准" |
| 包裹堆叠上限 | 3个 | 2-5 | 防止过高导致不稳定 | 视觉上清晰可见 |
| 堆叠稳定性 | 70% | 50-85% | 3层时70%不倒 | 有紧张感但可控 |
| 角色碰撞弹性 | 0.4 | 0.2-0.6 | 有击退但不失控 | 喜剧效果，非惩罚 |
| 包裹重量影响 | 速度-15% | 10-25% | 携带大件有代价 | 明显感知但可接受 |
| 重力加速度 | 15 m/s² | 12-20 | 比现实略快，更刺激 | 包裹下落有"重量感" |

#### 物理手感测试清单

**移动测试**：
- [ ] 角色能在3秒内从房间一端跑到另一端
- [ ] 急停时没有滑步感
- [ ] 转身180°响应即时

**投掷测试**：
- [ ] 最小力度投掷能精准命中1米外目标
- [ ] 最大力度投掷能跨越房间
- [ ] 抛物线轨迹可见且可预测
- [ ] 包裹落地有"重量感"（适当反弹后静止）

**碰撞测试**：
- [ ] 两人对撞会产生明显但无害的位移
- [ ] 包裹碰撞堆叠有"摇摇欲坠"的视觉效果
- [ ] 倒塌时有连锁反应的满足感

### 3.4 高级配合机制（新增）

#### 3.4.1 抛接系统
**设计目的**：增加动作层面的配合，创造"完美配合"时刻

| 动作 | 输入 | 成功条件 | 效果 |
|------|------|----------|------|
| 抛出 | 抓取时按投掷键 | - | 包裹进入"可接"状态（1.5秒） |
| 接住 | 在落点按抓取键 | 时机±0.3秒 | 无停顿直接处理，+15金币奖励 |
| 失败 | 错过时机 | 包裹落地 | 正常拾取流程，无惩罚 |

**抛接应用场景**：
- 跨越障碍物的快速传递
- 2人搬运大件时的位置调整
- 急件的最后冲刺接力

#### 3.4.2 同步操作
**设计目的**：强制2人配合，创造沟通需求

| 操作 | 场景 | 机制 | 沟通需求 |
|------|------|------|----------|
| 双开门 | 某些通道 | 需2人同时按住交互键 | "我数123一起按" |
| 紧急制动 | 失控传送带 | 2人同时拉下开关 | "快去右边开关！" |
| 协同扫描 | 大件包裹处理 | 1人搬运，1人协助扫描 | "别动，我扫一下！" |

#### 3.4.4 大件包裹处理流程（状态机）

```
[生成] → [待处理] → [搬运中] → [扫描确认] → [装车]
            ↓           ↓
         [滞留区]   [协助扫描]
                      ↓
                   [效率加成]
```

**详细流程**：

| 状态 | 说明 | 网络同步要点 |
|------|------|--------------|
| **生成** | 传送带生成大件包裹 | 主机生成，RPC通知所有客户端 |
| **待处理** | 包裹在传送带上等待 | 位置由主机同步（10Hz） |
| **搬运中** | 玩家抓取，移速-40% | Authority转移给持有者 |
| **协助扫描** | 另一玩家可协助扫描 | 需距离<2m，触发协作判定 |
| **效率加成** | 协作完成+20%金币奖励 | 主机计算，RPC通知奖励 |
| **装车** | 放入货车 | 主机验证，更新订单状态 |

**协作判定条件**：
1. 搬运者在扫描台范围内
2. 协助者在搬运者2m范围内
3. 协助者按住交互键1秒
4. 同时满足以上条件触发协作奖励

#### 3.4.3 角色技能（可选后期内容）
**设计原则**：不改变核心玩法，只提供微小优势

| 技能 | 效果 | 冷却 | 适用场景 |
|------|------|------|----------|
| 冲刺 | 3秒内移速+50% | 30秒 | 急件最后冲刺 |
| 稳定手 | 5秒内不会掉落易碎品 | 45秒 | 易碎品密集时 |
| 扫描眼 | 瞬间显示所有包裹标签 | 60秒 | 混乱分拣时 |

**获取方式**：等级解锁或装备系统，不影响平衡性

## 四、社交互动设计

### 4.1 强制沟通设计

**信息分割**：
- 订单信息只显示在特定屏幕（如1P看订单，其他人看不到）
- 包裹标签小字，必须靠近才能看清
- 货车目的地标识分散在不同位置

**角色互补**：
- 没有固定职业，但关卡设计鼓励分工
- 例如：传送带区需要1人，分拣台需要1-2人，装车区需要1人

### 4.2 正向社交反馈

| 触发条件 | 反馈效果 |
|----------|----------|
| 完美配合（如空中接力） | 慢动作回放 + "完美配合！" |
| 救场行为（如接住掉落包裹） | 特殊音效 + 额外金币 |
| 连续成功 | 连击计数器，氛围升温 |
| 搞笑失误 | 自动截图提示 + 回放标记 |

### 4.3 非对称信息关卡示例

**关卡："通讯中断"**
- 地图中间有堵墙分隔两队
- 1P和2P在A区看到订单信息
- 3P和4P在B区操作设备
- 必须通过语音沟通才能完成

## 五、经济系统与进度

### 5.1 货币系统

#### 5.1.1 金币获取（平衡后）

| 行为 | 基础奖励 | 加成条件 | 设计理由 |
|------|----------|----------|----------|
| 完成普通订单 | 30金币 | 提前完成+10金币 | 降低基础值，鼓励效率 |
| 完成急件 | 60金币 | 无超时+20金币 | 风险与回报比例 2:1 |
| 完美配合 | 15金币 | - | 锦上添花，非主要收入来源 |
| 无失误奖励 | 100金币 | 整局无包裹损坏 | 高难度挑战奖励 |
| 救场行为 | 10金币/次 | 上限30金币/局 | 鼓励互助，但防止刷分 |

**单局理论最大值计算**：
- 假设10分钟局内完成：8个普通订单 + 3个急件
- 基础收入：8×30 + 3×60 = 420金币
- 理想加成：8×10 + 3×20 + 30(救场) + 100(无失误) = 270金币
- **单局上限：690金币**（高手完美表现）
- **单局平均：250-400金币**（普通玩家）

#### 5.1.2 金币消耗（调整后的价格曲线）

| 物品类型 | 价格区间 | 解锁时机 | 设计理由 |
|----------|----------|----------|----------|
| 基础皮肤（3款） | 800-1200金币 | 等级1可购买 | 第3-5局可解锁首个皮肤 |
| 进阶皮肤（5款） | 1500-2500金币 | 等级5解锁 | 需要10-15局积累 |
| 稀有皮肤（3款） | 4000-6000金币 | 等级10解锁 | 长期目标，约20-30局 |
| 表情动作 | 200-500金币 | 等级2解锁 | 小额消费，即时满足 |
| 特效装扮 | 800-1500金币 | 等级7解锁 | 中期消费点 |

**经济健康检查公式**：
```
平均单局收入 × 目标局数 = 目标物品价格
300金币 × 6局 = 1800金币（首个进阶皮肤）
```

#### 5.1.3 防通胀设计

- **每日首局加成**：+50%金币（鼓励每日登录，控制总产出）
- **软上限机制**：同一行为重复超过阈值后收益递减
- **消耗出口**：定期活动/限时皮肤提供金币消耗出口

### 5.2 经验与等级

| 等级 | 所需经验 | 累计经验 | 解锁内容 | 预计局数 |
|------|----------|----------|----------|----------|
| 1 | - | - | 基础关卡、新手教程 | 1局 |
| 2 | 100 | 100 | 表情动作商店 | 1局 |
| 3 | 150 | 250 | 进阶教程 | 2局 |
| 4 | 200 | 450 | 港口关卡 | 2局 |
| 5 | 250 | 700 | 易碎品机制、进阶皮肤 | 3局 |
| 6 | 300 | 1000 | 成就系统 | 3局 |
| 7 | 400 | 1400 | 机场关卡、特效装扮 | 4局 |
| 8 | 500 | 1900 | 大件机制 | 5局 |
| 9 | 600 | 2500 | 排行榜 | 6局 |
| 10 | 700 | 3200 | 危险品、稀有皮肤预览 | 7局 |

## 六、新手引导设计

### 6.1 分层教学

**第一层：基础操作（第1关）**
- 强制暂停教学：移动、抓取、放置
- 无时间压力，无限重试
- 必须成功3次才进入下一步

**第二层：简单配合（第2-3关）**
- 引入2人配合（如大件搬运）
- 轻度时间压力
- 失败无惩罚，鼓励尝试

**第三层：完整体验（第4关起）**
- 完整订单流程
- 正常难度曲线
- 可随时查看操作提示

### 6.2 引导原则

- [x] 核心动词30秒内引入
- [x] 第一次成功有保障
- [x] 新机制在安全环境引入
- [x] 玩家通过探索发现机制
- [x] 第一局以"再来一次"钩子结束

## 七、技术规格

### 7.1 渲染方案

#### 7.1.1 渲染器选择
- **推荐方案**：Forward+ 渲染器
  - 原因：适合中等规模场景（同屏4角色+多包裹），支持多光源，性能可控
  - 备选：Mobile渲染器（如果目标配置较低）

#### 7.1.2 视觉效果规范
- **包裹效果**：使用简单颜色变化+发光（Emission），避免复杂后处理
  - 易碎品：淡蓝色微光
  - 急件：红色脉冲发光
  - 危险品：橙色警告光
- **事件效果**：UI遮罩为主，减少全屏后处理开销
  - 停电事件：圆形遮罩+暗角，非光照修改
- **着色器提示**：所有材质参数添加 `hint_range` 和 `source_color`
  ```glsl
  uniform vec4 base_color : source_color = vec4(1.0);
  uniform float emission_strength : hint_range(0.0, 2.0) = 0.5;
  ```

#### 7.1.3 美术风格确认
- **明确3D Low Poly**：非2D，使用低多边形+平面着色
- **目标帧率**：60fps（PC标准配置）
- **同屏面数预算**：< 50,000 三角面

### 7.2 网络架构

#### 7.2.1 网络拓扑（Star拓扑 - 主机-客机模式）

```
        [主机玩家 - Server]
           /    |    \
          /     |     \
    [客机1]  [客机2]  [客机3]
    (Client) (Client) (Client)
```

- **非真P2P**：所有逻辑以主机为权威（Authoritative Server）
- **Steam联机**：使用Steam P2P API建立连接，但逻辑仍是主机-客机模式
- **延迟容忍**：
  - < 150ms：流畅体验
  - 150-300ms：可玩，轻微延迟感
  - > 300ms：不推荐继续游戏

#### 7.2.2 权限分配表（Multiplayer Authority）

| 实体类型 | Authority归属 | 说明 |
|----------|---------------|------|
| 玩家角色 | 各客户端 | 本地预测 + 服务器校正 |
| 包裹（被持有） | 持有者客户端 | 跟随玩家移动，服务器验证 |
| 包裹（地面） | 主机 | 物理计算、位置同步 |
| 传送带 | 主机 | 生成逻辑、位置控制 |
| 订单系统 | 主机 | 生成、计时、完成判定 |
| 事件系统 | 主机 | 触发、状态管理 |
| 货车 | 主机 | 发车时机、装载判定 |
| 得分/金币 | 主机 | 最终权威，客户端显示 |

#### 7.2.3 物理同步策略

| 计算类型 | 执行位置 | 同步方式 | 频率 |
|----------|----------|----------|------|
| 角色移动 | 客户端预测 + 主机验证 | RPC + State | 60Hz输入，20Hz同步 |
| 包裹投掷 | 客户端发起 → 主机计算 | RPC触发 | 事件驱动 |
| 包裹地面物理 | 主机 | State同步 | 10Hz（优化带宽） |
| 碰撞检测 | 主机权威 | 触发事件RPC | 事件驱动 |
| 堆叠稳定性 | 主机 | State同步 | 5Hz（仅变化时） |

**关键优化**：
- 包裹被玩家持有时，使用 `remote_transform` 而非物理同步
- 投掷瞬间由主机计算轨迹，客户端插值显示
- 静止包裹进入"睡眠"状态，停止同步直到被唤醒

#### 7.2.4 防作弊设计

| 验证项目 | 服务器规则 | 违规处理 |
|----------|------------|----------|
| 移动速度 | 不得超过 4.5 m/s × 1.2（容错） | 校正位置，警告日志 |
| 投掷距离 | 不得超过 7.0 m | 拒绝该次投掷RPC |
| 金币获取 | 验证订单完成状态 | 拒绝非法增加 |
| 包裹抓取 | 验证距离 < 2.0 m | 拒绝抓取请求 |
| 订单完成 | 验证包裹类型、目的地匹配 | 拒绝非法完成 |
| 事件触发 | 仅主机可触发事件 | 忽略客户端事件请求 |

#### 7.2.5 断线处理

| 场景 | 处理策略 | 恢复机制 |
|------|----------|----------|
| 客机断线 | 角色变为AI托管或消失 | 支持重连，恢复游戏状态 |
| 主机断线 | 触发主机迁移（投票选新主机） | 迁移后游戏继续 |
| 临时断线（< 5秒） | 角色冻结，等待重连 | 自动恢复 |
| 长期断线（> 30秒） | 踢出玩家，AI接管或移除 | 无法恢复 |

**主机迁移流程**：
1. 检测主机断线（心跳超时5秒）
2. 剩余玩家投票选择新主机（按玩家ID最小优先）
3. 新主机加载游戏状态（从最近快照恢复）
4. 通知所有客机重新连接
5. 游戏继续（最多暂停10秒）

**状态快照策略**：
- 每30秒保存一次完整游戏状态
- 包含：所有包裹位置/状态、当前订单、得分、事件状态
- 主机迁移时从最近快照恢复

### 7.3 平台适配

- 支持键盘 + 手柄（混合输入）
- 本地分屏：2人上下分屏，4人四宫格
- 远程同乐：通过Steam实现

## 九、待验证假设

| 假设 | 验证方式 | 成功标准 |
|------|----------|----------|

| 抛接机制增加配合乐趣 | 抛接使用率统计 | 每局平均3次以上成功抛接 |
| 动态事件节奏合适 | 玩家压力自评 | 7/10玩家认为"刚好有挑战" |

### 10.2 游戏测试检查清单

**第一次原型测试（MVP阶段）**：
- [ ] 基础移动/抓取手感是否跟手？
- [ ] 2人配合搬运是否自然？
- [ ] 10分钟时长是否合适？
- [ ] 失误时玩家反应是笑还是骂？

**第二次测试（加入经济系统）**：
- [ ] 金币获取速度是否符合预期？
- [ ] 解锁第一个皮肤需要多少局？
- [ ] 玩家是否有明确的消费目标？

**第三次测试（完整体验）**：
- [ ] 动态事件是否增加乐趣？
- [ ] 抛接机制是否被使用？
- [ ] 4人配合是否混乱但可控？
- [ ] 新手引导完成率是否 > 90%？

### 10.3 崩溃测试场景

**测试游戏在什么情况下会"崩溃"（失去乐趣）**：

| 场景 | 预期崩溃点 | 预防措施 |
|------|------------|----------|
| 4个新手玩家 | 完全混乱，无法完成任何订单 | 新手关卡降低难度30% |
| 1个高手+3个新手 | 高手包揽一切，新手无事可做 | 设计必须多人配合的机制 |
| 连续触发3个事件 | 玩家 overwhelm，放弃治疗 | 事件触发间隔限制 |
| 网络延迟300ms+ | 抛接/配合机制失效 | 延迟容忍设计（±0.3秒窗口） |
| 玩家故意捣乱 | 游戏体验被破坏 | 投票踢人机制（后期） |
